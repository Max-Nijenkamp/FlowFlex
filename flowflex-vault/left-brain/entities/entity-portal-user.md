# Entity: PortalUser

External users who access one of the Vue+Inertia portals. Separate from the `users` table, which holds internal company team members who use the Filament `/app` workspace panel. PortalUsers have no access to the Filament workspace panel at any time.

**Table**: `portal_users`
**Multi-Tenant**: Yes â€” every row has `company_id`.

---

## Portal Types

Each PortalUser belongs to exactly one portal type, which determines which frontend portal they access and what permissions they can hold.

| portal_type | Portal URL | Who |
|---|---|---|
| `employee` | `/my/` | Company employees accessing the Employee Self-Service portal |
| `client` | `/portal/` | External clients viewing invoices, projects, tickets |
| `b2b` | `/b2b/` | B2B partner companies accessing a shared order/quote portal |
| `partner` | `/partner/` | Resellers and referral partners |
| `learner` | `/learn/` | External learners enrolled in LMS courses |
| `member` | `/community/` | Community members (only type that allows self-registration) |

## How PortalUsers Are Created

PortalUsers are never self-registered (except `member` type). They are created automatically by domain module logic:

| portal_type | Created By |
|---|---|
| `employee` | Auto-created when an HR Employee record is created, if the ESS portal is active for the company |
| `client` | Auto-created when a CRM Contact has "portal access" toggled to enabled |
| `b2b` | Created when a B2B company account is configured in the B2B module |
| `partner` | Created in the Partner module when a partner account is set up |
| `learner` | Created when a contact is enrolled in an external-facing LMS course |
| `member` | Self-registration via the Community portal â€” the only type with a public signup form |

---

## Invite Flow

1. A domain module creates a `portal_users` record with `status = invited`, `password = null`, and `invited_at = now()`.
2. An invite email is dispatched containing a one-time magic-link token (valid 7 days).
3. The user clicks the link, which opens the portal login page with their email pre-filled and a password-set prompt.
4. User sets their password: `email_verified_at` is set, `status` changes to `active`, portal session is created.
5. If the token expires, the owner can trigger a re-invite from the workspace panel, which generates a new token and invalidates the old one.

## Business Rules

1. **Email uniqueness**: Email must be unique per `company_id` + `portal_type` combination. The same email address can hold both a `client` record and a `learner` record for the same company â€” they are separate accounts.
2. **Invited status blocks login**: A `portal_users` record with `status = invited` cannot authenticate. The password is null and the login form rejects the attempt with "please check your email to set your password."
3. **Deletion event**: Soft-deleting a PortalUser dispatches a `PortalAccessRevoked` event. Domain modules listen to this event to clean up any related data (e.g. ESS revokes employee self-service access, LMS removes active course enrolments).
4. **portal_type is immutable**: Once set on creation, `portal_type` cannot be changed. A company employee who is also an LMS learner has two separate PortalUser records.
